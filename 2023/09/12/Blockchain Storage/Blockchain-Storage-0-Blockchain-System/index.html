论文链接：区块链的数据管理技术综述

区块链的核心构成

区块链网络中包含互不信任的多个节点，节点间无法确定某个节点是否为恶意节点。区块链网络假定某些节点可能产生恶意攻击，但网络中大多数节点是诚实节点。区块链中的节点大致可以分为三类：

  • 全节点：同步区块链上的全部数据，包括各个区块的哈希值、事务列表等
  • 轻节点：不保存区块链上的所有数据，往往只存储区块头部的的哈希值。有时会提交针对区块链上的数据的查询，并利用返回的结果以及自身储存的哈希值对结果进行验证
  • 矿工节点：特殊的全节点，不仅存储区块链上的所有数据，而且参与区块链的共识过程

区块链结构与事务

区块链网络节点存储整个区块链系统的数据以及事务的日志，数据以区块的形式进行存储并链接。区块链的结构如下图所示，区块链包括前一个区块的哈希值(PreHash)、共识验证字段(ConsProof)、默克尔哈希树根的哈希值(MerkleRoot)以及事务日志所构造的所构成的默克尔哈希树。

默克尔哈希树是一种二叉树，其叶子节点保存事务，非叶子节点保存子节点内容的哈希值，其特点为：改变任意一个对任何数据的变动，都会影响从叶子节点到根节点路径上的所有节点的哈希值，因此该结构可以用于验证数据是否被篡改。

区块链不允许数据修改现有区块，而是只允许在区块尾部添加新的区块。这使得区块链会记录所有对数据的更改操作。

区块链中一个事务对应一次智能合约的完成，其需要保证ACID：原子性(Atomicity), 一致性(Consistency), 隔离性(Isolation), 持久性(Durability)。

共识机制

由于区块链网络中节点不可信且要求所有节点均存储数据的相同状态。当其中一个节点的数据准备更新时，所有节点需对更新操作达成共识。共识协议往往基于两种思路：基于计算的共识协议和基于通信的共识协议。

基于计算的共识协议代表是比特币所使用的工作量证明(PoW, Proof of Work)，其原理为：节点通过工作量证明确定下一个有权力将生成的区块添加到链中的节点。挖矿节点根据区块内信息计算出该区块的共识验证字段ConsProof，使得其满足hash(PrevBKhash∣ConsProof∣MerkleRoot)≤Zhash(PrevBKhash|ConsProof|MerkleRoot)\le Zhash(PrevBKhash∣ConsProof∣MerkleRoot)≤Z，其中ZZZ是一个控制工作量大小的参数。最先计算出公式验证字段的挖矿节点将其结果全网广播，其它挖矿节点验证收到的验证字段是否满足条件，若满足，则将区块添加到已有区块链的尾部，并给最先计算出结果的节点相应的奖励。工作量证明可以很好的用于公有链网络中，可以抵御女巫攻击。

基于通讯的共识协议将投票权分配给网络中的节点，并利用多轮通讯达成共识。代表为使用拜占庭容错协议(PBFT)。PBFT共识协议需要经过三个过程

  1. pre-prepare: 用户提交请求时，一个主节点会在该阶段广播其请求
  2. prepare：所有节点收到请求后判断是否执行该操作，同意执行则将自己的签名添加在信息中并广播，否则不发送信息。节点收到其他节点发送的执行操作信息达到一定数量后，则说明执行达成共识。
  3. commit：节点执行相应请求，并将执行结果返回主节点。

现在有其它的协议在两种思路下完成取舍，如Elastico、RapidChain、Byzcoin、Tendermint等结合了两种方式。

基于计算的工作量证明共识并不能确定网络中的事务均合法，当恶意节点控制全网33%的算力时，则恶意节点有可能控制这个网络。而PBFT共识是确定性的，但其网络开销达到O(n2)O(n^2)O(n2)，网络产生的额外开销会超过增加算力所节省的实践，是的系统的性能随算力增加而降低，降低了网络的可扩展性。

UTXO与Account模型

区块链系统中，典型的数据模式是比特币所基于的Unspent Transactions Outputs(UTXO)结构，在UTXO模型中，不存在一个账户记录一个用户所拥有的所有比特币，而是使用UTXO包含比特币，交易是通过已有的UTXO生成新的UTXO完成，一个用户所拥有的UTXO是其拥有的所有UTXO之和。每个UTXO只会被用于一次交易，交易完后会生成包含等量比特币的新UTXO，所以大多数交易只需要判断一个UTXO是否被多次交易即可，因而交易是否进行可以很快的被验证，且具有很高的隐私性。但由于不存在账户的概念且交易基于UTXO，编程复杂度较高。

另一种数据模式是Ethereum所采用的Account模型，该模型保存一个用户所拥有的账户的所有可用电子货币。当一个交易被提交时，相应的事务首先判断账户内电子货币是否足以完成该交易，如果可以则执行。Account模型节省了大量的存储空间，而UTXO模型可以更好的保护用户隐私。

智能合约

智能合约是一段区块链所有节点都认可的代码，并存储在区块中（防止被篡改），当智能合约的条件被满足时，合约会被自动激活并执行相应代码。例如执行转账的智能合约会先判断其激活条件是否被满足激活条件，然后判断事务是否可执行，包括是否被篡改，账户金额是否足够等；验证合法后自动执行转账的事务。

区块链类型

根据节点加入或退出区块链系统的方式，可以分为三类

  • 公有链系统：也称非许可链，允许任何节点参与区块链数据维护和读取；同时，节点也可以随时加入或离开该网络。公有链系统是一个完全的去中心化系统，典型的有比特币、以太坊等。公有链的共识机制一般是PoW，理论上恶意节点需占有网络中51%的算力才可以控制该区块链。
  • 联盟链：也称许可链，新加入节点需要获得许可的区块链，联盟链限制可参与的成员节点，且节点的读写权限以及所负责的计算由联盟链的设计规则确定。典型的有Hyperledger Fabric。联盟链由所有节点共同维护，且假定网络中部分节点是可信的，节点的加入以及共识的过程一般通过可信的节点控制。联盟链多采用PBFT等基于通讯的共识算法。
  • 私有链：仅限信任的个体使用，且不完全能够解决信任问题。网络节点间彼此透明，但不对外公开。

区块链的数据存储

针对区块链的存储模式主要集中在两个方向，一个是基于键值对的存储模式，另一个是关系型数据的存储模式。

基于键值对的数据存储模式

许多区块链系统采用基于键值对的文件系统存储其数据状态，这样的存储方式往往具有较高的查询性能，Hyperledger Fabric均采用键值对数据库存储数据。

在典型的区块链系统中，各个节点均要存储区块链内数据的备份，这会导致数据在链上存储的代价极高；同时，考虑到部分共识协议需要大量网络开销，在数据规模极大的情况下，直接将所有数据存储在区块链中将产生巨大的存储开销和网络通讯代价。因此，部分工作旨在以较小的代价提高数据存储规模，包括分布式存储和链下存储等技术。

分布式存储

文献：Off-Chaining models and approaches to off-chain computations

实现思路：将区块链的节点分为多个区域，并且保证每个区域内所有节点数据可以通过整合生成整个区块链的数据。这样的情况下，单一节点就不需要再保存整个区块链内数据副本，降低了节点的存储开销，具有更好的拓展性。

实现方式：将数据分割为 nnn 个不同的块，并将其分配到一个区域的 mmm 个节点中。可以利用分布式存储的编码技术（如MDS编码）生成分配方案，使得再 mmm 个节点中任意获取 kkk 个节点，即可获得所有原始数据。

一致性保证：只要不存在超过 kkk 个恶意节点在同一个区域内，就仍可保持数据一致性。

数据隐私保护：通过Shanmir’s secret sharing技术加密，并将密钥分配给一个区域内的各个节点。各个节点密钥均不相同，当需要完整数据时，系统可通过各个节点的密钥计算出原始密钥，再对数据进行解密。

额外开销：因为需要对数据进行编码，当数据为动态时，由额外的编码计算开销

链下存储

文献：Decentralized iot data management using blockchain and trusted execution environment., Blockchain based data integrity service framework for iot data.

实现：

  • 将数据以键值对 <key,value><key,value><key,value> 的方式存储，原始数据往往直接存储在相应的云服务器中，云服务器中的数据不在区块链中，即在链下。而区块链中只需要存储相应数据的哈希值，即 <key,hash(value)><key,hash(value)><key,hash(value)> 。当用户通过云服务器获得数据时，即可利用哈希值进行验证，保证数据的完整性和一致性。
  • 由于从链上存储数据以及部署智能合约均需要存储开销，文献还提出了多种基于链下存储的优化技术。在计算方面可以将状态检查等计算采用链下验证的方式，而不需要采用链上的智能合约，降低存储开销。同时有文献提出智能合约在有需要的情况下也可以存储在链下。为了保证安全性，链下的执行环境需为可信环境；同时，区块链存储相应的智能合约状态的哈希值。在这样的结构下，网络同时存在计算节点和共识节点，计算节点负责智能合约的状态改变以及计算等工作，共识节点只负责对智能合约的状态哈希形成共识。

历史版本的存储

文献：Forkbase: An efficient storage engine for blockchain and forkable applications.

部分区块链操作需获取不同历史版本的数据，Forkbase设计了支持数据版本查询的数据存储系统，其将数据拆分为块，并在块的级别减少重复的数据冗余存储。其利用POS-Tree(pattern-oriented-split tree)这一索引快速确定重复数据。

POS-Tree的每个叶子节点包含对应的数据，而非叶子节点则保存其子树的键值；同时，利用默克尔树的特性，对每个节点计算其哈希签名，内部节点签名由其子节点签名决定。因此，当一个分支的根节点没有变化时，可以确定该节点的所有子节点均未更新版本。所以查找不同的数据时，只需要从签名不同的根节点开始，搜索其子树中带有不同签名的节点即可。

基于关系型数据的存储模式

文献：Blockchaindb— Towards a shared database on blockchains., Blockchaindb—A shared database on blockchains.

BlockchainDB针对共享表类型数据，并在多个用户均需要对共享表进行操作的情况下，支持对表进行数据验证。该系统采用联盟链的设计方式并假定节点不可信，区块链作为数据的存储层，保证数据在不可信的环境下的一致性。节点间存在多个分块，每一块独立地维护一个区块链并存储共享表的部分数据，每一块独立的区块链保证了分块内的数据一致性。

在每一块的区块链存储层上构造BlockchainDB数据库层，负责

  • 数据库层记录表记录任意一条存储记录在哪一些分块中
  • 提供用户可直接调用的API
  • 保证用户提交的事务处理一致性：当需要进行读写操作时，用户提交数据所在分片ID，BlockchainDB会在相应的分片中执行读写操作并返回结果给用户。用户获得数据后，需要验证所得到的数据是否被恶意篡改及其完整性。作者提供了两种验证方法
    • online验证：用户获得数据后广播所读数据并要求所有存储该数据的节点进行验证，如果大部分节点返回消息表示需要验证的数据与本地相符，则认为数据已经被验证
    • offline验证：延迟验证，当用户提交验证请求且验证请求达到系统设定的参数e时，BlockchainDB开始验证
