处理未捕获异常的处理器

线程的run方法不能抛出任何检查型异常，如果其中出现了未捕获的异常，将导致线程死亡，在死亡前，异常会传递到一个用于处理未捕获异常的处理器。

这个处理器实现了Thread.UncaughtExceptionHandler接口，该接口只有一个方法：

如果没有安装处理器，则默认处理器为null；如果没有为单个线程安装处理器，那么处理器是该线程的ThreadGroup对象，它的uncaughtException方法执行以下操作：

  1. 如果该线程有父线程组，那么调用父线程组的uncaughtException方法
  2. 否则，如果Thread.getDefaultExceptionHandler返回一个非null处理器，则调用该处理器
  3. 否则，如果Throwable是ThreadDeath的一个实例，什么都不做
  4. 否则，将线程名字和Throwable的栈轨迹输出到System.err

可重入锁ReentrantLock

可重入锁基本使用：

这个结构确保了只有一个线程进入临界区。一旦一个进程锁定了锁对象，其它任何进程都无法通过lock语句。

这种锁称为重入锁，因为线程可以反复获得已拥有的锁，支持对锁的嵌套调用，每一次调用lock后都需要unlock释放。由于这个特性，被一个锁保护的代码可以调用另一个使用相同锁的方法。

await方法会将该线程放入这个条件变量的等待集中，即使锁可用了也不会唤醒，而是等待一个同一条件的signalAll方法：

这个调用会重新激活等待这个条件的所有线程，使其变为RUNNABLE状态。同时线程会尝试重新进入该对象，一旦锁可用，则从await返回，得到这个锁，并从之前暂停处继续执行。

但signalAll方法并不保证条件被满足，它仅仅是一个通知，线程应当自行检查条件是否被满足。

Lock/Condition与synchronized

建议：

  • 最好不要使用Lock/Condition机制也不使用synchronized关键字。在许多情况下，可以使用java.util.concurrent包中的某种机制处理所有的锁定。

volatile字段

如果为了读写一两个字段而使用同步，其带来的性能开销是不划算的，但不使用同步又可能带来错误。volatile提供了一个免锁机制，字段被声明为volatile后，编译器和虚拟机就知道该字段可能被另一个线程并发更新。

除了volatile字段，声明为final的字段也可以安全的并发访问

线程安全的集合

线程安全的集合指能够在多线程并发访问的情况下保证不被破坏的集合

阻塞队列BlockingQueue

阻塞队列在队列的基础上，特有的方法：

方法 行为 特殊情况下的行为
put 添加一个元素 队列满时阻塞
take 移除并返回队头 队列空时阻塞
add 添加一个元素 队列满时抛出IllegalStateException
element 返回队头元素 队列空时抛出NoSuchElementException
offer 添加一个元素并返回true 队列满时返回false
peek 返回队头元素 队列空时返回null
poll 移除并返回队头 队列空时返回null
remove 移除并返回队头 队列空时抛出NoSuchElementException

阻塞队列，根据处理特殊情况的方法不同，阻塞一般有三种用法：

  • 线程管理工具：使用put和take
  • 希望抛出异常时：使用add、remove、element
  • 希望只得到错误提示：使用offer、peak、poll

BlockingQueue本身是一个抽象类，但java.util.concurrent包中提供了几个其实现：

  • LinkedBlockingQueue：容量没有上界（但可以给出参数指定容量）
  • LinkedBlockingDeque：双端队列
  • ArrayBlockingQueue：需要在构造时指定容量

还有PriorityBlockingQueue，其为一个支持阻塞特性的优先队列

高效的映射、集和队列

java.util.concurrent提供了一系列集合在并发情况下的高效实现：ConcurrentHashMap、ConcurrentSkipListMap、ConcurrentSkipListSet和ConcurrentLinkedQueue。这些集合用复杂的算法允许并发地访问数据结构的不同部分尽可能减少竞争。

但这些集合的size方法不一定能在常数时间内完成操作，通常需要遍历才能得到当前大小

集合返回弱一致性的迭代器，意味着迭代器不一定能反映出构造之后的所有更改，但不会返回同一个值两次，也不会抛出异常

ConcurrentHashMap支持原子更新，可以通过以下方法实现：

  • compute(key, (k, v) -> {...})
    • 同理还有computeIfPresent和computeIfAbsent用来更新key存在/不存在时的更新
  • merge(key, newVal, (oldVal, newVal) -> {...})

同时ConcurrentHashMap还支持批操作，批操作可以遍历映射并进行处理，包含以下三种操作：

  • search：为每个key或value应用一个函数，知道函数生成一个非null结果返回
  • reduce：组合所有的key或value，使用一个累加函数处理
  • forEach：为所有的key或value应用一个函数

每种操作都有四个版本：处理key、处理value、处理key和value、处理Map.Entry对象，例如search的四个方法如下：

  • searchKeys
  • searchValues
  • search
  • searchEntries

任务与线程池

构造一个新线程的开销有些大，如果程序中创建了大量生命期短的线程，那么不应该把任务映射到一个单独的线程，而应该使用线程池。

线程池中包含许多准备运行的线程，为线程池提供一个Runnable就会有一个线程调用run方法。当run方法退出时，线程池中的线程不会死亡，而是留在池中等待下一个请求。

控制任务组

执行器另一个强大的功能是可以控制一组相关的任务，常用方法如下：

  • shutdownNow：取消所有任务

  • invokeAny：提交一个Callable对象集合中的所有对象，并返回某个（往往是最早完成的）已完成任务的结果。这对于接受任一答案的搜索问题很有用。

  • invokeAll：提交一个Callable对象集合中的所有对象，阻塞直到所有任务完成，返回所有任务的Future对象列表
